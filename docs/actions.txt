-  Image Generation — Zero Budget Fix:
Use Unsplash Source API. No key, no billing, no signup. Make sure you use the right API or URL for the call so it does not fail
https://source.unsplash.com/400x400/?skincare,women
https://source.unsplash.com/400x400/?streetwear,clothing
https://source.unsplash.com/400x400/?jewelry,handmade
The trick is — you pass keywords from the user's prompt directly into the URL. So when someone says "skincare brand for Gen Z women in Accra" you extract skincare, women, beauty and the images are always relevant, always high quality, always free, no rate limit that matters at MVP scale.
Implementation:
jsfunction getProductImage(productName, storeCategory) {
  const keywords = extractKeywords(productName, storeCategory)
  return `https://source.unsplash.com/400x400/?${keywords.join(',')}`
}

No fal.ai, no pollinations, no picsum randomness. Relevant images from day one. Ship it.

-  Dynamic Stores — The Real Fix
The problem isn't Next.js. It's that you're using a fixed template with fixed slots. Lovable works because every prompt produces genuinely different component trees.
What to do:
Stop thinking "template + data." Start thinking "prompt → layout config → render."
Your AI call should return not just products but a store schema:
{
  "layout": "editorial",
  "colorScheme": "warm-minimal",
  "heroStyle": "full-bleed",
  "sections": ["hero", "featured", "grid-6", "testimonial"],
  "productCount": 6,
  "typography": "serif-led",
  "products": [...]
}

Then you have 3–4 layout variants that render differently based on that schema. A jewelry boutique gets serif fonts, warm tones, editorial layout. A streetwear store gets bold type, dark theme, grid layout. Same codebase, genuinely different stores.
That's the dynamic feel without rebuilding everything. Claude picks the schema, your components render it.

When the store is created and canonical is saved, run one extra Groq call to classify it. Add one field to your Postgres schema:
sqlALTER TABLE stores ADD COLUMN layout_variant VARCHAR(20) DEFAULT 'grid';
-- values: 'editorial' | 'grid' | 'bold'
The classifier prompt is tiny and fast:
const classifyLayout = async (canonical) => {
  const prompt = `
You are a store layout classifier.
Given this store description, return ONLY one word: editorial, grid, or bold.

Rules:
- editorial: luxury, handmade, jewelry, art, beauty, skincare, fashion boutique
- bold: streetwear, sports, tech, gaming, sneakers, urban, men's fashion  
- grid: general, food, home, gifts, books, kids, multi-category

Store name: ${canonical.store_name}
Store description: ${canonical.description}
Category: ${canonical.category}
Products: ${canonical.products.map(p => p.name).join(', ')}

Reply with one word only.`

  const result = await groq.chat(prompt)
  return result.trim().toLowerCase() // 'editorial' | 'grid' | 'bold'
}

Save it to the canonical:
canonical.layout_variant = await classifyLayout(canonical)
await db.stores.update({ layout_variant: canonical.layout_variant }, { where: { id: store.id } })

Step 2 — The 3 Variants Defined
Each variant is a different component composition — same data, different feel.

editorial — Luxury / Boutique:
"- Full-bleed hero image, store name in serif
- Products in 2-column asymmetric grid
- Large product images, minimal text
- Warm or neutral palette extracted from category
- Good for: skincare, jewelry, handmade, fashion"

grid — Clean / Marketplace:
"- Standard hero with tagline + CTA
- Products in 3-column even grid (or 2 on mobile)
- Card-based with price prominent
- Neutral palette, high contrast
- Good for: general stores, food, gifts, home"

bold — High Energy / Street:
"- Dark hero, large typography, accent color
- Products in staggered 2-column with hover effects
- Price badges, bold product names
- Dark palette with one strong accent
- Good for: streetwear, sneakers, sports, tech"

Step 3 — Next.js Rendering
Your store page reads the variant and switches the component tree:
// pages/store/[slug].jsx

import EditorialLayout from '@/layouts/EditorialLayout'
import GridLayout from '@/layouts/GridLayout'
import BoldLayout from '@/layouts/BoldLayout'

const LAYOUTS = {
  editorial: EditorialLayout,
  grid: GridLayout,
  bold: BoldLayout,
}

export default function StorePage({ canonical }) {
  const Layout = LAYOUTS[canonical.layout_variant] || GridLayout

  return <Layout store={canonical} />
}

export async function getServerSideProps({ params }) {
  const canonical = await db.stores.findOne({ 
    where: { slug: params.slug } 
  })
  return { props: { canonical } }
}


Step 4 — Unsplash Images from Canonical
Each product already has a name in the canonical. Use that plus the store category:
function getProductImage(productName, storeCategory) {
  const clean = (str) => str.toLowerCase().replace(/[^a-z0-9 ]/g, '').trim()
  const keywords = [
    ...clean(productName).split(' ').slice(0, 2),
    clean(storeCategory).split(' ')[0]
  ].join(',')
  
  return `https://source.unsplash.com/600x600/?${keywords}`
}

// Usage inside layout component
canonical.products.map(product => ({
  ...product,
  image: getProductImage(product.name, canonical.category)
}))

1. User types: "Launch a skincare brand for Gen Z women in Accra"
2. Groq generates canonical → saved to Postgres
3. Classifier runs → layout_variant: "editorial" → saved
4. Image keywords extracted per product → saved
5. Store page loads → reads canonical → picks EditorialLayout
6. Products render with Unsplash keyword images
7. Paystack checkout wired to your test account

Every store feels different. Zero extra cost. No new dependencies.
